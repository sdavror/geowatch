## Built round 3 (`feature/baltic-registries`) — beneficial-ownership registries, not sanctions lists

Same RU/UA/BY-scoping convention as every sanctions source, applied to a different kind of data: instead of "is this entity itself designated," these ask "does this company have a Russian/Ukrainian/Belarusian beneficial owner" — a real ownership-network signal for shell-company/sanctions-evasion research, distinct from a designation. Individual beneficial owners (real people) are never modelled as Entities, same reasoning as Companies House PSC.

| Source | Country/scope | Type | Auth | Result |
|---|---|---|---|---|
| Estonia e-Business Register | Estonia | beneficial ownership | keyless bulk download | Free JSON bulk file (`kasusaajad.json.zip`, self-contained — company name + beneficial owners in one record, no join needed). **Real bug found live**: this Cloudflare-fronted host returns 403 specifically to Node's native `fetch()` (undici) while `wget`/`curl` from the exact same container get 200 — confirmed by testing both from inside the running container against the identical URL; adding browser-like headers to fetch() didn't help either, pointing to a TLS/HTTP-client fingerprint block rather than a UA or network-path issue. Fixed by shelling out to `wget` for this one download. Live run: **4,422 processed → 4,422 new entities** (1,176 of the underlying 374k-company file have an RU-resident beneficial owner alone). Verified live (`2Circas OÜ`, correct `reg_number@EE`). |
| Latvia Enterprise Register | Latvia | beneficial ownership | keyless bulk download | Two separate free CSVs (beneficial owners + main register) requiring a two-file join — the owners file only has a registration number, not a company name. Owners file carries clean ISO2 `nationality` *and* `residence` (richer than Estonia's residence-only). **Real bug found live**: the 128MB register.csv download died mid-transfer with undici's `TypeError: terminated` on the first live run (silent connection drop, not an HTTP error) — fixed with the same fetch-with-retry shape as the OFSI adapter. Live run: **3,862 processed → 3,862 new entities**. Verified live (`SIA Tour Up in Europe`, correct `reg_number@LV`). |

## Built round 4 (`feature/global-registry-expansion`) — general on-demand registries beyond the EU/English-speaking world

The user's stated end goal is genuinely global coverage (explicitly called out Japan as an example gap). This round's 4 priority candidates (Japan, Brazil, Denmark, South Korea) all hit real, documented obstacles — but two adjacent Nordic candidates (Norway, Finland) turned out to be excellent free/keyless on-demand registries, same shape as Companies House/France SIRENE.

| Source | Country/scope | Type | Auth | Result |
|---|---|---|---|---|
| Norway Brønnøysund Register (Enhetsregisteret) | Norway | registry, on-demand search-by-name | keyless | Free, keyless, real-time REST API (`data.brreg.no`) — confirmed live 2026-07-20. One search call gives name/address/industry/**website** (a field most other sources lack entirely); a separate `/roller` endpoint gives real reported board members and daily manager (e.g. verified against Equinor ASA — CEO "Anders Opedal" as `Daglig leder`, plus board). Status normalized from three real boolean fields (`konkurs`=bankrupt→liquidated, `underAvvikling`/`underTvangsavviklingEllerTvangsopplosning`→dissolved). Best general-registry source found so far — better field coverage than Companies House. |
| Finland PRH YTJ v3 | Finland | registry, on-demand search-by-name | keyless | Free, keyless, real-time REST API (`avoindata.prh.fi`) — confirmed live 2026-07-20, name search and businessId lookup both work. Gives website, industry code **with an English-language description** (`mainBusinessLine.descriptions`), and address. **Deliberately left `status` unset**: the API returns numeric codes (`status`, `tradeRegisterStatus`) with no documented enum found anywhere in the public docs — guessing a mapping risked silently wrong data, so this field stays null pending a real source for the code meanings (same "don't invent meanings for unclear fields" discipline as the France/Companies House status mapping). No officer/director data exposed by this API, unlike Norway's `/roller`. |
| Switzerland Zefix (Central Business Names Index) | Switzerland | registry, on-demand search-by-name | keyless (undocumented frontend API) | The **official documented API** at `zefix.admin.ch` requires a paid subscription (confirmed live: 401 without a key). Instead uses the `www.zefix.ch` frontend's own internal API — undocumented, but confirmed live 2026-07-20 with no key/auth at all. Real UID (Swiss registration-number equivalent), registered canton/commune, and status (`EXISTIEREND`→active, `IN_AUFLOESUNG`→dissolved — both confirmed by searching a real defunct company, Swissair's various liquidating entities). Deliberately sparse: Zefix is a name INDEX, not the full commercial register — no street address, industry code, website, or officers. Worth building despite the gaps given Switzerland's real-world relevance as a common holding/trading jurisdiction. Caveat: being undocumented, this endpoint could change without notice. |

## Built round 5 (`feature/registry-expansion-round2`) — top-economy OWN sanctions regimes

Direction change per the user: prioritize top economies' OWN independent sanctions programs (not a copy of/overlap with US/EU/UK), by real-world impact — not a continuation of RU/UA/BY-only scoping. Same live-verification discipline as every prior source.

| Source | Country/scope | Type | Auth | Result |
|---|---|---|---|---|
| Japan MOF Sanctions List | Japan | sanctions | keyless | Japan's own asset-freeze designation program (Foreign Exchange and Foreign Trade Act), not a UN/US/EU copy. One consolidated CSV, free, keyless. **Real risk avoided**: the download link is relative and dated (changes every revision) — the adapter re-resolves it from the live page every run instead of hardcoding a filename, avoiding the exact mistake WebFetch's own URL extraction made on the first attempt. Live run: **850 processed → 850 new entities** (no structured identifiers, same shape as Canada/Australia — relies on Phase 2/3 fuzzy+LLM). Real cross-source candidates surfaced correctly (North Korean shipping/trading entities scoring against existing OFAC-derived records). |
| Switzerland SECO Sanctions List | Switzerland | sanctions | keyless (undocumented mirror host) | Switzerland's own sanctions program. **Real finding**: this exact source was marked "confirmed dead, NXDOMAIN" in a prior session — that check hit `sesam.search-admin.ch` (hyphen); the real host is `sesam.search.admin.ch` (dot), a one-character difference between a dead domain and a live 40MB consolidated XML list. The XML format nests full historical `<modification>` blocks inside each `<target>` (each containing its OWN nested `<target>` sub-copy for audit purposes) — naive full-document target-counting would multiply-count the same entity's history; parses only the top-level array and determines "currently listed" from the latest modification's type. **Real bug found live**: the 40MB download hit the same undici "TypeError: terminated" dropped-connection failure as Latvia's 128MB CSV — same fetch-with-retry fix. Live run: **8,604 total targets → 2,496 currently-listed organizations** (5,321 individuals + 729 de-listed correctly excluded). |

Researched, blocked this round: **China** (MOFCOM Unreliable Entity List / Export Control List) — confirmed no consolidated structured dataset exists at all; designations are only published as individual press-release announcements in free-form Chinese, not a downloadable CSV/XML/API, and the ministry's site partially 403s automated access. **South Korea** (MOEF financial sanctions list) — designations are published one-at-a-time as dated bulletin-board notices (each its own page, attachments likely HWP/PDF not CSV), not a single current consolidated file the way Japan's MOF publishes one; the bulletin list page itself is also JS-rendered (no notice titles visible to a plain fetch). **Singapore** (MAS Targeted Financial Sanctions) — confirmed via MAS's own page: Singapore has no consolidated list of its own at all, just links to the UN Security Council resolutions plus a small TSOFA schedule with no downloadable file found (no PDF/CSV/XLSX links on the page). All three share the same shape: real designations exist, but not as a structured, keyless, single-file dataset like Japan/Switzerland — would need HTML-scraping fragile bulletin systems or aggregating many individual notices, a materially different (and much lower-confidence) build than every other source in this project.
